No Sense of Time

The schedule view may look great, but aesthetics aren’t enough; it currently lacks any visual indication of when each session starts, or how long it’s likely to last. You’ll fix this now.

In this short challenge, you’ll add some headers to the schedule layout that display the hours of the day.

Lab: The Session Starts When?

Since collection views can have pretty much any style of layout, the concept of headers and footers doesn’t really apply here, especially when you’re subclassing UICollectionViewLayout directly, instead of UICollectionViewFlowLayout. So in their place, Apple has provided supplementary views, which can be associated with each cell.

A collection view can contain many different types of supplementary views, and you provide each one with a unique string identifier, known as its kind, so that you can infer which type you’re dealing with when in the data source, delegate, and layout methods.

Each cell can have many supplementary views associated with it, as long as its kind is unique within the scope of that cell – that is, you can’t have multiple supplementary views of the same kind associated with a single cell.

Even though you can do far more than just headers with supplementary views, in the case of this layout you are going to use supplementary views to create headers. Specifically, you’ll create a series of supplementary view headers that display the hours of the day, and have the custom layout position them horizontally across the top of the view.

Open ScheduleLayout.swift and add the following just below where you declared widthPerHour:

private let hourHeaderHeight: CGFloat = 40

This simply holds a fixed value representing the height of the headers.

Next, just below where you declared sessionFrames, add the following:

var hourHeaderFrames: [Frame] = []

Just like with the sessions, you’re going to calculate the frames and index paths of the headers in advance of layout, and cache them.

Now you need a method that can actually calculate the frame of a header at a given index path: Add the following just below frameForSession(_:atIndexPath:):

private func frameForHourHeaderViewAtIndexPath(indexPath: NSIndexPath) -> CGRect {  
  let frame = CGRect(x: widthPerHour * CGFloat(indexPath.item), y: 0, width: widthPerHour,  
    height: collectionViewContentSize().height)  
  return frame  
}

The math looks a lot more complicated that it actually is - you take a predetermined value for the width of the header and multiply it by the item of the index path to work out the x coordinate. The y coordinate is 0, since you want the header fixed to the top of the view. The width is the predetermined value, and the height is the same as the height of the collection view - that might strike you as being a little odd, but all will become clear shortly.

Whilst you’re dealing with the utility methods, you need to make a small change to frameForSession(_:atIndexPath:) so it takes the height of the header into account. Find the following statement:

let y = heightPerTrack * CGFloat(indexPath.item)

And replace it with this one:

let y = hourHeaderHeight + (heightPerTrack * CGFloat(indexPath.item))

This makes sure that the height of the header is now taken into account when calculating the frames of the sessions in the schedule.

With everything now in place to calculate and cache the frames for the headers, add the following to the bottom of prepareLayout():

// 1  
if hourHeaderFrames.isEmpty {  
  // 2  
  for hour in 0..<dataSource.numberOfHoursInSchedule {  
    // 3  
    let indexPath = NSIndexPath(forItem: hour, inSection: 0)  
    // 4  
    let rect = frameForHourHeaderViewAtIndexPath(indexPath)  
    // 5  
    let frame = Frame(rect: rect, indexPath: indexPath)  
    hourHeaderFrames.append(frame)  
  }
}

Here’s what’s happening:

  1. As the schedule is static it’s far more efficient to only execute this code the first time prepareLayout() is called, since it’s not only called when the collection view is first presented, but also whenever its layout is invalidated.
  2. Iterate over the number of hours in the schedule.
  3. Create an index path for the current hour; these index paths are different to those used by the cells. You can think of cells and supplementary views operating in different layers with independent index paths.
  4. Calculate the frame of the header at the current index path.
  5. Wrap the frame and index path using the Frame struct and cache it.

The final step to adding the headers is to actually create the layout attributes for each one that’s currently on screen. Add the following to layoutAttributesForElementsInRect(_:), just above the return statement:

// 1  
for frame in hourHeaderFrames {  
  // 2  
  if CGRectIntersectsRect(frame.rect, rect) {  
    // 3  
    let attributes = UICollectionViewLayoutAttributes(forSupplementaryViewOfKind: "HourHeader",  
      withIndexPath: frame.indexPath)  
    // 4  
    attributes.frame = frame.rect  
    // 5  
    attributes.zIndex = -1  
    // 6  
    layoutAttributes.append(attributes)  
  }
}

Here’s what’s happening:

  1. Iterate over each cached header frame.
  2. Check to see if the header’s rect intersects with the rect provided to the method by the collection view.
  3. If it does, create a set of layout attributes for the supplementary view, using HourHeader as its kind.
  4. Set the frame of the header to the cached frame.
  5. Set the zIndex to -1 so that the header is guaranteed to appear underneath the session cells, which is necessary for this layout, as you’ll see later.
  6. Append the layout attributes to the layoutAttributes array.

With all that done, build and run.

Wait…what’s that? Oh, it’s crashed? Take a look at the error message printed to the console:

'could not dequeue a view of kind: HourHeader with identifier ScheduleHeader - must register a nib or a  
 class for the identifier or connect a prototype cell in a storyboard'

Ah, of course! You’ve told the collection view to use a supplementary view with the kind HourHeader for the header, but you’ve not actually registered a matching class or nib for it to use as the UI.

Open ScheduleViewController.swift and add the following to viewDidLoad() just below the call to super:

let headerNib = UINib(nibName: "ScheduleHeader", bundle: nil)  
collectionView!.registerNib(headerNib, forSupplementaryViewOfKind: "HourHeader",  
  withReuseIdentifier: "ScheduleHeader")

This tells the collection view to use the nib named ScheduleHeader for any supplementary view with the kind HourHeader. The nib is already part of the project so you don’t need to worry about creating that yourself.

The nib is pretty basic, containing a label and a thin gray view used as a vertical separator. The Custom Class is set to ScheduleHeader, which is a subclass of UICollectionReusableView; all supplementary views must descend from this class so they can be properly managed by the collection view.

Build and run. Fantastic! No more crashing, and the calendar now has headers.

You can see the headers are laid out properly across the top of the collection view. You’ll also notice the faint hour line that runs from the top to the bottom of the view; this is the reason why the height of the header was set to match the height of the collection view, and why the zIndex was set to -1, because it needed to run underneath the cells.

The final task is to give the headers a proper title, in this case the hour each section represents.

Open ScheduleViewController.swift and add the following to viewDidLoad() just below the cell configuration block:

dataSource.headerConfigurationBlock = {(header: ScheduleHeader, indexPath: NSIndexPath, group: Group,  
  kind: String) in  
  header.titleLabel.text = dataSource.titleForHourHeaderViewAtIndexPath(indexPath)  
}

Here you add a header configuration block to the data source. The data source calls this block once for each header that is to be displayed in the collection view. You use it to set the titleLabel of the header to the title for the hour at the given index path, using a convenience method on the data source.

Build and run. Your headers now have a proper title. Nice work!
