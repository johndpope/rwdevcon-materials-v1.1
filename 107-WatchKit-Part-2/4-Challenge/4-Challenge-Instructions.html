107: WatchKit II, Part 4: Challenge Instructions

At this point you’ve built a pretty solid WatchKit App.

In Greg’s session you created an attractive interface using tables to display the sessions, and page-based interface controllers to display the session and presenter details.

In this session you’ve added a context menu that allows you to favorite a session, you’ve added a Glance which displays your next favorite session, and finally you’ve implemented Handoff so the WatchKit App displays the session details of the session being shown by the Glance, when launched via the Glance.

But there is an elephant in the room. You may have noticed that when you favorite a session in the WatchKit App, that change isn’t reflected in the My Schedule tab in the iOS app.

This is happening because the two apps don’t share a single data source; instead they each have their own Core Data store seeded using the RWDevCon2015.plist file.

Well that’s daft! Why don’t you simply share the same data source?

Good question! And the answer’s a simple one; so you can fix the issue using a different approach, and along the way learn about some really cool functionality provided by the WatchKit framework. The cynical amongst you might even think Greg and I specifically planned it that way :]

The Challenge

Your challenge, should you choose to accept it, is to make sure any change made to a session’s favorite status on the WatchKit App is reflected in the My Schedule tab on the iOS app.

In the WatchKit App you’ll need to use the class method WKInterfaceController.openParentApplication(_:reply). This launches the containing iOS app in the background and calls a method on its app delegate. You supply a userInfo dictionary, which is passed to the iOS app, and a reply block, which is called when the iOS app replies.

In the iOS app you’ll need to override the application(_:handleWatchKitExtensionRequest:reply) method in the app delegate. This method is passed the userInfo dictionary supplied by the WatchKit App, and a reply block which you call when you’re finished, and which accepts a replyInfo dictionary that’s then passed back to the WatchKit app.

Here are some hints to get you started:

  • toggleFavorite() in DetailsInterfaceController.swift seems as good a place as any to launch the iOS app
  • The iOS app will need a way to identify the session that’s been (un)favorited
  • It probably also needs to know the current favourite status of that session
  • It’d be nice if the iOS app replied stating whether or not it was able to mirror the change on its end

Make sure to give this a go yourself before reviewing the solution.

Bug Alert: As of Beta 4 you can no longer run both the WatchKit app and the iOS app at the same time (you could in previous betas), which makes debugging the iOS app whilst the WatchKit app is running, and vice-versa, almost impossible. Hopefully Apple will fix this in a future released.

You have 15 minutes…good luck!

The Solution

Open DetailsInterfaceController.swift and add the following inside the if block in toggleFavorite(), just below configureContextMenu() :

WKInterfaceController.openParentApplication(["identifier": session.identifier, “isFavorite": session.isFavorite],  
    reply: { (replyInfo, error) -> Void in  
  if let success = replyInfo["success"] as? Bool {  
    println("Favorite status for '\(session.title)' synced with iOS app: \(success)")  
  }
})

Here you call openParentApplication(_:reply) passing both the identifer and favorite status of the current session via the userInfo dictionary. In the reply block you extract the success boolean from the replyInfo dictionary and print it to the console.

Next, open AppDelegate.swift in the RWDevCon group and add the following just below application(_:didFinishLaunchingWithOptions:):

func application(application: UIApplication!, handleWatchKitExtensionRequest userInfo: [NSObject : AnyObject]!,  
  reply: (([NSObject : AnyObject]!) -> Void)!) {  
  // 1  
  if let identifier = userInfo["identifier"] as? String {  
    // 2  
    if let isFavorite = userInfo[“isFavorite"] as? Bool {  
      // 3  
      if let session = Session.sessionByIdentifier(identifier, context: coreDataStack.context) {  
        // 4  
        session.isFavorite = isFavorite  
        // 5  
        reply(["success": true])  
        return  
      }
    }  
  }
  // 6  
  reply(["success": false])  
}

Here’s a breakdown of what’s happening:

  1. Extract the session identifier from the userInfo dictionary
  2. Extract the favorite status from the userInfo dictionary
  3. Fetch the corresponding session from the Core Data store using the identifier
  4. Update the favorite status of the session so it matches the passed favorite status. You don’t simply toggle the status like you do in the WatchKit App as the two aren’t guaranteed to be in-sync to begin with
  5. Call the reply block stating the update was successful
  6. If anything goes wrong, call the reply block stating the update failed

Now, select the RWDevCon scheme and build and run. Make sure the My Schedule tab is empty, and then stop the app.

If it’s not empty, select each session and tap Remove from My Schedule.

Next, select the RWDevCon WatchKit App scheme and build and run. Select Keynote from the list, and then long-press when the session details are presented to invoke the context menu. Tap the Favorite menu item.

Note: If the session was previously favorited, you’ll need to unfavorite it first, and then favorite it again for this step to work

After the menu is dismissed, you should see the following printed to the console:

Favorite status for ‘Keynote’ synced with iOS app: true

Finally, relaunch the iOS app (simply do this in the simulator, you don’t need to change the scheme or build and run) and then open the My Schedule tab. Behold as the session is now a favorite on the iOS app too :]

If you got this far, congratulations! You’ve finished the challenge. And not only do you have a great WatchKit App and a great iOS app, but the two are now kept in-sync as well - or at least, they are in one direction :]
